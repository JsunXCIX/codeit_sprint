# SQL JOIN의 이해와 실무적 활용: 데이터의 결합과 가치 창출

## 1. SQL에서 JOIN이 왜 필요한가?
데이터베이스는 효율적인 관리를 위해 정보를 주제별로 나누어 저장한다. 하지만 개별 테이블 하나만 봐서는 복합적인 비즈니스 인사이트를 도출하기 어렵다.

### 데이터의 시너지 효과
예를 들어 피자 메뉴(종류, 가격) 테이블과 판매 기록(시간, 수량) 테이블이 각각 있다고 가정해보면, 각 테이블이 가진 정보의 가치를 점수로 주관적으로 매겨보면 각각 1점 정도일 것이다.

### 1 + 1 = 100 이상의 가치
하지만 `JOIN`을 통해 두 데이터를 결합하면 "점심시간에 가장 잘 팔리는 피자는 무엇인가?" 또는 "요일별 매출 추이"같은 고부가가치 정보를 생성할 수 있다. 이때 데이터의 가치는 단순한 산술 합계인 2점이 아니라, 100점, 1000점 이상의 가치로 뻥튀기된다.

> 결론: JOIN은 파편화된 데이터를 연결해 의미 있는 비즈니스 언어로 변환하는 필수적인 연산이다.

## 2. INNER JOIN vs LEFT JOIN: 실무적 차이와 선택 기준

결합 방식에 따라 결과 집합이 완전히 달라지므로, 분석 목적에 맞는 선택이 중요하다.



### ① LEFT JOIN (부분적 정보 허용)
왼쪽 테이블을 기준으로 오른쪽 테이블을 붙이는 방식이다. 기준이 되는 왼쪽 데이터는 무조건 유지된다.

* 특징: 오른쪽 테이블에 매칭되는 값이 없으면 `NULL`로 표시된다.
* 실무 예시: '전체 회원별 리뷰 작성 현황 조회'
    * 왼쪽(A): 전체 회원 명단 / 오른쪽(B): 리뷰 작성 내역
* 활용: 리뷰를 쓰지 않은 회원도 명단에 나와야 할 때 사용한다. 리뷰 칸이 `NULL`인 회원을 타겟으로 리뷰 작성 이벤트 push 알림을 보내는 기획안을 작성할 수 있다. 즉, 누락되면 안 되는 핵심 리스트가 있을 때 선택한다.

### ② INNER JOIN (교집합 데이터만 추출)
두 테이블 모두에 존재하는, 즉 공통된 데이터만 살리는 방식이다.

* 특징: 한쪽이라도 데이터가 없으면 결과에서 완전히 제외된다.
* 실무 예시: '실제 판매된 상품의 상세 스펙 분석'
    * 테이블(A): 판매된 주문 건 / 테이블(B): 전체 상품 마스터
* 활용: 판매되지 않은 상품 정보나, 주문 기록이 없는 유령 상품 정보는 분석에서 제외하고 싶을 때 사용한다. 오직 두 데이터가 모두 존재해야만 의미가 있는 분석(예: 실제 매출이 발생한 데이터의 통계)에 유용하다.
